# Codex × Claude 連携案（コーディング外部委譲）

> ステータス：**提案段階（未導入）**。実装はまだ。別PCでも参照できるよう本ファイルに記録。
> 最終更新：2026-06-12

## 目的
コードの**生成（書く作業）が最もトークンを消費する**ため、そこを ChatGPT Pro の **Codex** に委譲し、Claude は「設計・レビュー・ドキュメント・進捗管理」に集中してトークンを節約する。

## 目指す形（VS Code ターミナル1枚で完結）
ユーザーは **Claude Code（1ターミナル）にだけ指示**する。
Claude が裏で：仕様を書く → `codex` を Bash 経由で起動して実装させる → `git diff` をレビュー → ドキュメント＆コミット。
→ Codex の画面を別で開く必要なし。コーディングのトークンは Codex（ChatGPT Pro）側へ。

## 役割分担
| 工程 | 担当 | 理由 |
|------|------|------|
| 要件・設計・**API契約／TypeScript型の確定** | Claude | 仕様の“正”を一元管理し、両者の衝突を防ぐ |
| 実装（`app/` のコード） | **Codex** | 生成トークンを Codex 側へ逃がす |
| レビュー（`git diff` を読む） | Claude | 読む＜書く で相対的に安い |
| ドキュメント・進捗・コミット文面 | Claude | テキスト主体で軽い |

ポイント：**インターフェース（APIの形・型）は Claude が先に固定**し、Codex は中身を埋めるだけにする。

## 運用ループ（1指示あたり）
1. Claude：要件 → 仕様＋型/API契約を確定（`app/` のインターフェースの正は Claude が握る）
2. Claude：実装を Codex に委譲（コードは Codex が書く）
3. Claude：`git diff` をレビュー → 必要なら 2 へ差し戻し
4. Claude：ドキュメント更新＋**段階ごとにコミット＆プッシュ**（author=Orica256・Claude表記なし）

## セットアップ作業案（後で実施）
**フェーズ0：前提確認（最重要）**
- Codex CLI をインストールし、ChatGPT Pro でログイン。
- **非対話（ヘッドレス）実行が可能か確認**：プロンプトを渡すと確認なしでファイルを編集して終了するモードがあるか。
  - 無い／毎回承認が要る仕様なら「1ターミナル自動運用」は不可 → 手動リレー（Claude が仕様出力 → 人が Codex に貼る）に格下げ。

**フェーズ1：呼び出し経路**
- `codex` を Claude Code の許可リスト（`.claude/settings.json`）へ登録し、権限プロンプトを抑制。
- 薄いラッパースクリプト（例 `scripts/codex-task.sh`）：仕様ファイルを受け取り `codex` をヘッドレス起動するだけ。
- ハンドオフ用テンプレ：Claude が書く「作業指示（完了条件つき）＋API契約／型」を Codex がそのまま読める1枚に。

**フェーズ2：運用ループの固定化**（上記4ステップ）
- 検証（`tsc --noEmit`・起動・curl）は Codex 側に寄せ、Claude は結果ログのみ確認。

**フェーズ3：小さく試す**
- 1エンドポイント追加など極小タスクで1周し、トークンの出方・手戻り・競合を観察してから本採用。

## 決めておくこと
- 境界線：型/API契約は Claude 固定 → Codex は中身だけ、で良いか。
- レビュー深度：Claude は「差分だけ」か「起動確認まで」か。
- アクター：バック/フロントの“中の人”だけ Codex に置換（PM/レビュアー/秘書/ドキュメントは Claude）。
- コミット権限：**Claude 固定**を推奨（履歴の乱れ・著者統一のため）。

## リスク・確認事項
1. Codex CLI のヘッドレス対応可否（要実機確認）。NG なら手動リレー。
2. 同時編集の競合 → 必ず逐次（実装→レビュー）。並行させない。
3. 規約のズレ → Codex にも `CLAUDE.md`（無料/ローカル・命名規則）を毎回渡す。
4. レビューにも読みトークンは要る → 差分中心で最小化。
